## 📖 Описание проекта

Coffee Shop Management System — это система автоматизации работы кофейни, покрывающая полный цикл: от онлайн-заказа клиентом до аналитики продаж для владельца.

Проект не ограничивается типовым CRUD — в нём реализована бизнес-логика (расчёт стоимости заказа, программа лояльности, управление статусами заказа в реальном времени) и отдельный аналитический модуль с визуализацией данных.

**Целевые пользователи:**
- 👤 **Клиент** — регистрируется, просматривает меню, оформляет заказ, следит за статусом, копит бонусы.
- ☕ **Бариста / персонал** — видит очередь заказов, меняет статусы (принят → готовится → готов → выдан).
- 👑 **Администратор** — управляет меню, ценами, сотрудниками, смотрит аналитику продаж.

## 🏗 Архитектура

Проект построен по слоистой архитектуре (Layered Architecture), близкой к Clean Architecture:

```
Client (HTML/CSS/JS)
        │
        ▼
   API Layer (FastAPI routers, Pydantic-схемы)
        │
        ▼
  Service Layer (бизнес-логика: расчёт цены, бонусы, статусы)
        │
        ▼
Repository Layer (запросы к БД через SQLAlchemy)
        │
        ▼
   PostgreSQL
```

**Принцип:** каждый слой знает только о соседнем нижнем слое. API не обращается к БД напрямую — только через сервисы и репозитории. Это упрощает тестирование (можно мокать репозитории) и делает код читаемым для комиссии на защите.

```

├── src/

│   ├── schemas/               # Pydantic DTO
```

## 🎨 Frontend-часть (HTML/CSS/JS)

Поскольку фронтенд делается на чистом HTML/CSS/JS (без фреймворков), важно показать, что и без React/Vue можно сделать современный, живой интерфейс. Вот что реально сделать и как это выглядит по страницам:

### 1. `index.html` — Лендинг кофейни
- Hero-секция с описанием заведения, CSS-анимации при скролле (`IntersectionObserver`).
- Блок "Популярные напитки" — подтягивается через `fetch('/api/v1/products?featured=true')`.
- Адаптивная вёрстка (CSS Grid + Flexbox, медиа-запросы).

### 2. `catalog.html` — Каталог / Меню
- Динамическая загрузка категорий и товаров через API.
- Фильтрация по категориям на клиенте (без перезагрузки страницы) — фильтрация через JS по data-атрибутам.
- Поиск с debounce (`setTimeout` + `clearTimeout`) — не долбим API на каждое нажатие клавиши.
- Корзина — состояние хранится в `localStorage`, счётчик товаров обновляется в реальном времени в шапке.

### 3. `login.html` / `register.html`
- Форма с client-side валидацией (regex для email, длина пароля) до отправки на сервер.
- После логина — JWT токен кладём в `localStorage`/httpOnly cookie, дальше все запросы идут с `Authorization: Bearer`.
- Красивые состояния ошибок (неверный пароль, email занят) — без alert(), а через всплывающие toast-уведомления на чистом JS.

### 4. `order-tracking.html` — Трекинг заказа в реальном времени
Это один из самых эффектных элементов для защиты:
- Открывается WebSocket-соединение (`new WebSocket('wss://.../ws/orders/{id}')`).
- Статус заказа визуально меняется на прогресс-баре: Принят → Готовится → Готов → Выдан.
- Анимация прогресс-бара через CSS `transition` + JS переключение классов.
- Если сервер пришлёт "заказ готов" — показываем браузерное уведомление через `Notification API`.

### 5. `dashboard.html` — Аналитический дашборд (для администратора)
- Графики на **Chart.js**: линейный график выручки по дням, столбчатая диаграмма топ-5 товаров, круговая диаграмма долей категорий в продажах.
- Тепловая карта загруженности по часам дня (можно сделать на чистом CSS Grid + JS, раскрашивая ячейки по интенсивности).
- Карточки KPI сверху (общая выручка, число заказов, средний чек) — с анимацией "счётчика" на JS (число "накручивается" от 0 до значения при загрузке страницы).
- Данные обновляются без перезагрузки — periodic `fetch` каждые N секунд (или WebSocket, если хочется совсем "живой" дашборд).

### Общие технические приёмы, которые стоит показать
- Единый модуль `api.js` — обёртка над `fetch`, которая сама добавляет JWT-заголовок, обрабатывает 401 (редирект на логин) и парсит ошибки.
- Тёмная/светлая тема — переключение через CSS-переменные (`--bg-color`, `--text-color`) и `localStorage` для запоминания выбора.
- Skeleton-загрузчики (серые "заглушки" вместо спиннера) на время загрузки данных — выглядит очень современно и просто в реализации на CSS.
- Полностью адаптивная вёрстка — mobile-first, проверено на разных разрешениях.

## 📊 Визуализация и аналитика

Аналитический блок — то, что превращает проект из "ещё один CRUD" в систему для реального бизнеса. Реализуется в связке: SQL-агрегации на бэкенде → JSON API → Chart.js на фронтенде.

### Метрики, которые стоит показать
1. **Выручка по дням/неделям/месяцам** — линейный график, `GROUP BY DATE(created_at)`.
2. **Топ-5 самых продаваемых товаров** — столбчатая диаграмма, `GROUP BY product_id ORDER BY COUNT(*) DESC`.
3. **Доля категорий в продажах** — круговая диаграмма (напитки/десерты/еда).
4. **Средний чек** — KPI-карточка, `AVG(order.total)`.
5. **Загруженность по часам дня** — тепловая карта, помогает понять, когда нужно больше персонала.
6. **Динамика новых клиентов** — рост базы пользователей по неделям.
7. **Эффективность программы лояльности** — сколько бонусов начислено/списано, влияние на повторные заказы.

### Технически
- Все агрегации — на уровне SQL (оконные функции, `GROUP BY`), а не в Python-цикле — это важный момент для защиты (показывает понимание производительности БД).
- Эндпоинт `/api/v1/analytics/*` отдаёт готовый JSON под нужды графиков.
- Можно закэшировать тяжёлые агрегации в Redis на 5–10 минут, чтобы дашборд не долбил БД при каждом обновлении — это тоже хороший пункт для пояснительной записки ("оптимизация производительности").

## 🗄 Модель данных

**Основные сущности:**

| Таблица | Ключевые поля |
|---|---|
| `users` | id, email, hashed_password, role, bonus_points, created_at |
| `categories` | id, name, description |
| `products` | id, name, price, category_id, is_available, image_url |
| `orders` | id, user_id, status, total_price, created_at |
| `order_items` | id, order_id, product_id, quantity, price_at_order |
| `payments` | id, order_id, method, status |
| `loyalty_transactions` | id, user_id, points, type (earn/spend), created_at |
| `reviews` | id, user_id, product_id, rating, comment |

**Связи:** `User 1—N Order`, `Order 1—N OrderItem`, `Product N—1 Category`, `User 1—N LoyaltyTransaction`.

*(Рекомендуется приложить в пояснительную записку ER-диаграмму, сгенерированную через `dbdiagram.io` или `pgAdmin`.)*

## 🔐 Авторизация и роли

- JWT: access-токен (короткоживущий, 15–30 мин) + refresh-токен (долгоживущий, hardened cookie).
- Пароли хранятся только в виде bcrypt-хэша.
- Роли: `customer`, `barista`, `admin` — реализовано через enum в модели `User`.
- RBAC на уровне зависимостей FastAPI: `Depends(require_role("admin"))`.
- Опционально: OAuth2 через Google для быстрой регистрации клиентов.

## 🔌 API Эндпоинты (основные группы)

```
POST   /api/v1/auth/register
POST   /api/v1/auth/login
POST   /api/v1/auth/refresh

GET    /api/v1/products
POST   /api/v1/products          (admin)
PATCH  /api/v1/products/{id}     (admin)

POST   /api/v1/orders
GET    /api/v1/orders/{id}
PATCH  /api/v1/orders/{id}/status (staff)

GET    /api/v1/analytics/revenue
GET    /api/v1/analytics/top-products
GET    /api/v1/analytics/hourly-load

WS     /ws/orders/{order_id}
```

Полная спецификация доступна автоматически через Swagger UI: `http://localhost:8000/docs`

## 🚀 Установка и запуск

```bash
git clone <repo-url>
cd coffee-shop

cp .env.example .env
# заполнить переменные окружения (DB_URL, SECRET_KEY и т.д.)

docker-compose up --build
```

После запуска:
- API: `http://localhost:8000`
- Swagger: `http://localhost:8000/docs`
- Frontend: `http://localhost:3000`
- Дашборд: `http://localhost:3000/dashboard.html`

Локально без Docker:
```bash
python -m venv venv && source venv/bin/activate
pip install -r requirements.txt
alembic upgrade head
uvicorn src.main:app --reload
```

### Ближайшие улучшения

- [ ] Push-уведомления в браузере о готовности заказа

### Более амбициозные улучшения (для "выжать максимум")
- [ ] **Предсказание спроса** — простая ML-модель (даже линейная регрессия на исторических данных заказов), предсказывающая нагрузку на завтра — красиво ложится на дашборд как "прогноз".

- [ ] **Мультитенантность** — расширение системы на сеть кофеен (несколько точек, привязка заказов к конкретной точке).
- [ ] **PWA** — сделать фронтенд Progressive Web App (offline-режим для просмотра меню, установка на телефон).

- [ ] **GraphQL как альтернативный слой API** — для демонстрации владения разными подходами (опционально, не всем нужно).

### Технический долг / для раздела "недостатки" в диссертации
- Обработка гонок при одновременном списании остатков ингредиентов (нужны транзакции с блокировками).
- Нет пока полноценной retry-логики для фоновых задач.
- Кэш аналитики инвалидируется по TTL, а не по событию — можно улучшить через event-driven инвалидацию.

*(Наличие честного раздела "что не идеально и почему" — сильный сигнал зрелости для комиссии; лучше показать, что ты видишь ограничения, чем делать вид, что их нет.)*
